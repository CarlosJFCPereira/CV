Java 17 Spring Boot JavaFX React Native MySQL JWT Maven OpenPDF

Este projeto foi, de longe, o mais desafiador que já construí. Exigiu que pusesse em prática tudo o que aprendi ao longo destes anos — arquitetura de software, segurança, bases de dados, interfaces gráficas e desenvolvimento mobile — tudo ao mesmo tempo, num único sistema coeso. Não é um exercício académico nem um tutorial seguido à letra: é software real, pensado para resolver um problema real. É o projeto do qual mais me orgulho como programador e acredito que demonstra claramente as minhas capacidades de desenho e construção de sistemas complexos.

O que é o GadoApp?

O GadoApp é um sistema completo de gestão de explorações bovinas desenvolvido de raiz, com uma arquitetura multi-cliente e multi-tenant. Um único backend serve simultaneamente uma aplicação desktop JavaFX, uma app Android em React Native e uma página web estática — todos autenticados via JWT.

O sistema cobre o ciclo de vida completo de uma exploração: desde o registo de cada animal e a sua genealogia, passando pelo acompanhamento reprodutivo (inseminações, gravidezes, partos), saúde podal, gestão financeira e geração automática de eventos de calendário.

O Problema das Permissões — e Como o Resolvemos

O Problema

Um utilizador pode estar associado a várias explorações em simultâneo, com roles completamente diferentes em cada uma — por exemplo, Veterinário numa vacaria e Tratador de Cascos noutra.

O cliente JavaFX tratava as permissões como globais ao utilizador. Quando alguém iniciava sessão, o cliente não sabia em que exploração estava a operar e aplicava as permissões erradas, mostrando ou ocultando funcionalidades de forma incorreta.

A Solução

Tornar as permissões contextuais à exploração selecionada, em vez de globais ao utilizador. Quatro peças que trabalham em conjunto:

1
Endpoint dedicado no servidor

O servidor expõe um endpoint que devolve a matriz completa de permissões para aquela exploração específica, incluindo as flags de administrador e de proprietário.

2
Cache local por sessão

Ao selecionar uma exploração, o cliente carrega as permissões dessa vacaria e guarda-as em cache. Cada verificação de acesso consulta sempre esta cache local — sem pedidos extra ao servidor.

3
Fallback seguro

Se o servidor não responder, o sistema assume zero permissões por defeito — princípio do menor privilégio. Um aviso aparece na barra de navegação.

4
UI reativa

O menu principal oculta completamente os módulos sem permissão. As listagens desativam os botões de editar e eliminar se o utilizador não tiver as ações correspondentes naquele módulo.

A chave foi simples mas não óbvia: as permissões são sempre pedidas com o identificador da exploração atual. Mudar de exploração força um novo pedido ao servidor — e o cliente reconstrói o estado de permissões do zero.
✓
O resultado desta abordagem

Roles completamente modulares e independentes por exploração. Adicionar uma nova permissão ou um novo módulo é uma alteração de uma linha — o sistema propaga automaticamente para a UI de todos os clientes. Resolver o problema desta forma tornou o sistema extensível sem tocar na lógica de autenticação existente.
